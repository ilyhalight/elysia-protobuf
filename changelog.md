# 2.0.0

> [!CAUTION]
> This major release includes breaking changes
> Please review the migration notes before upgrading

## Breaking Changes

- `protobufParser`, `decode` decorator, `responseSchema` macro, and `schemas` option have been removed
- Protobuf now works through Elysia's standard `body` and `response` schemas
- Register `protobuf()` directly on the root `new Elysia()` instance, not only inside a `group`, so its `protobuf` parser is available to every route
- For routes with `parse: "protobuf"`, use `TProtobuf()` with a generated `ts-proto` schema in `body` to decode requests and in `response` to encode responses

> [!NOTE]
> Give `TProtobuf()` a schema generated by `ts-proto` from your `.proto` file, not a TypeBox schema

**Before**

```ts
const app = new Elysia()
  .use(
    protobuf({
      schemas: {
        "tags.request": RequestTags,
        "tags.response": ResponseTags,
      },
    }),
  )
  .post(
    "/a",
    async ({ body, decode, headers }) => {
      const data = await decode("tags.request", body, headers);
      return { tags: data.tags };
    },
    {
      parse: "protobuf",
      responseSchema: "tags.response",
    },
  );
```

**After**

```ts
const app = new Elysia()
  .use(protobuf())
  .post("/a", ({ body }) => ({ tags: body.tags }), {
    parse: "protobuf",
    body: TProtobuf(RequestTags),
    response: TProtobuf(ResponseTags),
  });
```

## Other Changes

- Signature verification now happens while parsing the request and is configured with `protobuf({ signature })`
- Missing or invalid signatures throw `ProtoRequestError` with status `400`
- A signature configuration inside a nested `group` isn't applied, so use the root plugin configuration instead
- `ProtoResponseError` has been removed
- Protobuf decoding and encoding failures now throw `ProtoValidationError` with status `422`
- Import error classes from `elysia-protobuf/error` rather than the package root
- When using `response: TProtobuf(...)`, return a plain object because a directly returned `Response` bypasses protobuf response encoding
- `signature.showParseErrors` controls whether `ProtoRequestError` parser messages are returned to the client. It defaults to `true`; set it to `false` to use Elysia's default parse-error response (Bad Request)

Example of custom error handling for parse errors:

```ts
.onError(({ code, error }) => {
  if (code === "PARSE" && error.cause instanceof ProtoRequestError) {
    return {
      error: error.cause.message,
    };
  }
})
```

- Requires Elysia `>= 1.3.0` for compatibility with TypeBox Transform system
